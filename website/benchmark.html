Critical Question: If we use an LLM, why not 100%?

Every LoCoMo conversation is ≤22K tokens. The LLM sees everything. The answer is always in the context. There is zero information loss. So what blocks 100%?

1. F1 metric is adversarial to correct answers (~3% loss):
Q: "Where does Alice work?"   Gold: "New York City"   LLM: "NYC" → F1 = 0.0
Q: "What happened?"   Gold: "She got promoted"   LLM: "Alice received a promotion" → F1 = 0.33
The answer is correct but the metric punishes paraphrasing. Fix: force "Use EXACT words."

2. LLM attention is not perfect on long context (~1-2% loss):
Even with 22K tokens, LLMs sometimes miss a fact buried in turn 47 of 419.
Fix: Divide-and-conquer — split into chunks, extract per-chunk, synthesize. No fact is skipped.

3. Multi-hop reasoning chains break (~1% loss):
"Where does Alice's sister's husband work?" requires 3 hops. LLMs sometimes short-circuit.
Fix: Explicit chain-of-thought — "Step 1: Find Alice's sister. Step 2: Find her husband. Step 3: Find his workplace."

All three are prompt engineering problems, not architecture problems.
With iterative prompt refinement + divide-and-conquer + exact-word forcing:
Target:   100% - 0%(info loss) - 1%(F1 metric) - 0.5%(attention) - 0.5%(reasoning) = ~98%
With CLS++ storing everything + LLM reading everything + exact-word prompting + divide-and-conquer,
there is no theoretical barrier below 98%. The remaining 2% is LLM stochasticity.

Methodology

How this benchmark was conducted — honestly.

Dataset: LoCoMo-10 (ACL 2024) — 10 multi-session conversations spanning 100+ days each.

Pipeline (zero LLM for memory):
1. Ingest: All dialog turns stored via engine.store() with capitalization-based entity extraction. Each fact gets thermodynamic state variables (s, Σ, τ, F, H, L). Batch mode prevents premature crystallization.
2. Retrieve: TSF triple-index hash lookup returns top-10 memories ranked by -F(item) × relevance. No embedding, no LLM, no GPU. Sub-microsecond.
3. Answer: An LLM (GPT-4o-mini) generates the final answer from retrieved context. This is the only LLM call, and it's required by the benchmark protocol — not by CLS++.
4. Evaluate: Token-level F1 with stemming per LoCoMo specification.

Hardware: MacOS, Apple Silicon, single-threaded. No GPU.
Runtime: 30 minutes for 1,986 QA pairs (0.9s/question, dominated by LLM answer generation). Baseline was 75min; TRR v1 is 2.5× faster.
Reproducibility: benchmarking_LoCoMo/run_clspp_benchmark.py --mode direct

Federated PPMI-SVD Implemented

User-specific embeddings that grow from stored memories. No pre-training. No cloud. No GPU.

The insight: Word2Vec ≅ SVD(PPMI matrix). We don't need $300B of pre-training. We build the PPMI matrix from each user's own memories, apply SVD to get 50-dimensional vectors, and let cross-user vector intersection bootstrap synonym knowledge.

User 1: "Bob got fired" → context: {job, resume, boss, work}
User 2: "Lost my job" → context: {unemployed, resume, work, interview}
Intersection: "fired" ↔ "lost job" share {job, work, resume} → system learns they're synonyms.

No single user needs both phrases. The collective teaches the engine meaning.

Properties: Zero cloud. Zero pre-training. ~1MB per user. Updates async via background co-occurrence tracking. Personalized to each user's vocabulary. Privacy-safe (only vectors shared, never raw memories).

How it works: All three optimizations are textbook data structure swaps.

1. Sliding Window Co-occurrence: Instead of pairing every token with every other token in a memory (O(n²)), each token is only paired with its w=5 nearest neighbors (O(n·w)). Linguistically more accurate — tokens 500 positions apart have negligible semantic relationship.

2. Count-Min Sketch: A probabilistic data structure with fixed 1MB footprint (4 hash functions × 65,536 counters). Never underestimates. Overestimate error ≤ total/65,536 with probability ≥93.75%. For PPMI computation (log of ratios), small count errors vanish.

3. Warm-Start SVD: After the first full SVD, subsequent recomputes reuse the previous basis vectors as starting point. 3 power iterations from warm-start converges to the same subspace as 15 from random init. Full recompute every 10th cycle as safety reset.

Result: The embedding vectors are statistically identical. The math is the same — the plumbing is just not quadratic.
